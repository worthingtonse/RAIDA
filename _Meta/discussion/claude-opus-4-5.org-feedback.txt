## Review of Gemini's Proposal (gemini.organize.txt)

**Strengths:**
- Very detailed and well-structured proposal
- Clear two-folder separation: Specification/ vs Reference/
- _Meta/ folder for working files is a good idea
- Detailed mapping table showing old->new locations
- Clear instruction for AI: "parse /Specification exclusively for protocol"

**Issues:**

1. **CRITICAL: 11_File_Formats is NOT protocol**
   Gemini puts `11_File_Formats/` into `Specification/03_Data_Formats/`.
   This is WRONG. File formats (.bin, .json, .stack) define how tokens are
   STORED ON DISK, not how they're transmitted on the wire. This is exactly
   the kind of confusion we're trying to prevent.

2. **CRITICAL: 14_Security is NOT protocol specification**
   Threat model and security checklists are GUIDANCE, not wire format.
   Putting it in Specification/ muddles the distinction.

3. **00_Introduction arguably IS protocol context**
   Moving glossary and architecture to Reference/ may be wrong. The glossary
   defines protocol terminology. Architecture explains the 25-RAIDA design
   which IS fundamental to the protocol (consensus rules, etc.).

4. **Too disruptive** - Complete restructuring breaks ALL existing cross-
   references between documents. Every link like `../07_Commands/` would break.

5. **Doesn't address duplicate 06_ prefix** - Both 06_Denominations and
   06_Terminator have same number.

6. **Renumbers everything** - Changing from 00-15 numbering to new 01-05
   numbering loses the familiar structure.
