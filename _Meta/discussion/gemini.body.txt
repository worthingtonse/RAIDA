This file details issues found in the documentation for RAIDA response headers, bodies, and status codes, based on analysis of the `RAIDAX-master` source code (`protocol.c` and `protocol.h`).

## 1. Response Body & Header Analysis

The primary issue is that the documentation does not clearly describe how the response is constructed, and it incorrectly implies the response body contains a challenge, which it does not.

### Issue 1: The Response Body Contains No Challenge/CRC

The user asked how the encryption type changes the body and if it has a challenge like the request.

**Finding:** The response body's structure is `[payload][trailer]`. It **does not** contain a Challenge/CRC structure like the request body. The server's "answer" to the client's challenge is the 16-byte `challenge_hash` placed in the **response header**.

-   **For Encryption Type 0:** The payload is sent in **plaintext**.
-   **For Encryption Types 1 & 2:** The payload is **encrypted** using AES-CTR. The server uses the same AN (key) and 8-byte nonce that the client provided in the request to encrypt the response payload.

**Recommendation:** The `00_Overview.md` file should be updated to explicitly state that the response body only contains the command output and does not have a challenge/CRC section.

### Issue 2: Incomplete "Signature" Calculation in `00_Overview.md`

The documentation states: `Signature = Challenge XOR Encryption_AN`. This is incomplete.

**Finding:** The `challenge_hash` (what the doc calls `Signature`) in the response header is generated differently based on the encryption type.

-   **For Types 1 & 2:** The formula `decrypted_challenge XOR AN` is correct.
-   **For Type 0:** There is no AN. The server simply **copies the 16-byte challenge** it received from the client's request body directly into the response header's `challenge_hash` field.

**Recommendation:** The "Signature Field" section in `00_Overview.md` must be updated to describe the behavior for both Type 0 and Types 1/2.

### Issue 3: Minor Errors in Response Header `00_Overview.md`

-   **Execution Time Unit:** The documentation claims the 4-byte execution time (`EX`) is in **nanoseconds**. The code calculates it in **microseconds**. This is a factual error.
-   **Field Name Mismatches:** The field names in the documentation's byte map (`RI`, `SS`, `CG`, `EC`, `SZ`, `EX`, `HS`) differ from the variable names used in the code (`raida_no`, `status`, `cgroup`, `e0/e1`, `output_size`, `exec_time`, `challenge_hash`). While not a critical error, aligning these would improve clarity.
